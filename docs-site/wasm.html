The promise made in Phase 5

The Design choices section ended with: "the natural successor to the TS implementation is a Penelope-implemented runtime — not a Rust rewrite that would be obsolete the moment the self-hosted compiler can target WASM or native."

Phase 6.A is the first concrete step toward that. std/wasm.pen is a WASM backend written in Penelope. It takes an AST bundle (the kind std/parser.pen produces) and emits a complete WebAssembly module as a list of bytes — valid binary, loadable by any WASM runtime, no external assembler required.

$ pen wasm examples/09-fib.pen --out fib.wasm
wrote fib.wasm (80 bytes)

$ node -e "
  const m = await WebAssembly.compile(require('fs').readFileSync('fib.wasm'));
  const i = await WebAssembly.instantiate(m);
  console.log('fib(20) =', i.exports.main());
"
fib(20) = 6765

Nine end-to-end tests (test/wasm-backend.test.ts) cover trivial literals, arithmetic, if/else, fn calls, recursive fib, factorial, mutual recursion (is_even/is_odd), bool-as-return, and a WebAssembly.validate round-trip. Each test runs the same source through both the pen-built WASM backend and the TS interpreter and asserts the results match.

What's out of scope (documented honestly)

Each item below is genuinely a separate, substantial project. None of them are MVP cuts dressed up as "future work" — they were excluded with eyes open because doing them right means a different design conversation.

ExcludedWhat it would take
StringsWASM linear-memory string layout (length-prefixed bytes), string builtins reimplemented against memory, possibly a string-interning table.
Lists / dictsHeap allocator in WASM memory; layout for variable-length sequences and hash buckets; GC strategy (mark-sweep or refcount).
Closures with capturesClosure-conversion pass: lift captured vars into a heap-allocated environment struct; pass an env pointer alongside the call.
First-class fn valuesWASM funcref + table + call_indirect; would also need type indices for every fn signature actually called indirectly.
Effects (print / fetch / ...)WASM imports for each host function; effect-log side table maintained in linear memory; the host runtime needs to vend the imports.
Snapshot / pause / resumeCPS transform on the AST so every function returns an explicit continuation; pause becomes "save continuation + heap, exit". This is a paper-sized change.
match expressionsPattern-compile-to-WASM equivalent of the current bytecode pattern compiler; or a structured-CFG recovery pass on existing bytecode.

If you need any of these, run the program through the regular interpreter or the JIT. The WASM backend is for the hot-path subset where you actually benefit from native execution — recursive math, bit-twiddling, decision trees over ints.

How it's built

The backend is one file, std/wasm.pen (~360 lines of Penelope), with no host helpers. It depends only on std/iter.pen (HOFs over lists). Structure:

  1. LEB128 encoders — leb128_u for unsigned (sizes, indexes), leb128_s_nonneg for signed-but-non-negative (the i32.const operand; negative literals are desugared to 0 - N by the parser, so this restriction is fine).
  2. Byte-buffer helpers — buffers are plain Penelope lists of ints. vec_of_bytes, name_bytes, str_to_bytes + an ASCII lookup table (Penelope has no char_code builtin).
  3. Two-pass compilation — pass 1 collects top-level fn-binding names and assigns indexes (incl. main); pass 2 emits each fn body's WASM instructions.
  4. AST → instructions — compile_expr walks the AST, recurses through BinOp / If / Block / Call / Var / literal nodes, and emits the corresponding WASM opcode bytes.
  5. Local-slot allocation — prebuild_env walks a fn body once to assign every let name a WASM local index. WASM locals are flat, so even let-bindings inside nested if branches share the single per-fn slot pool.
  6. Module assembler — wraps each section in (id, length, payload), glues them in spec order (type → function → export → code), prepends the \\0asm\\01\\00\\00\\00 header.

Why this matters

Before Phase 6.A, "Penelope can compile Penelope" meant "to Penelope bytecode that the TS VM runs." That's already a real fixpoint, but the bytecode lives or dies with the TS host.

Phase 6.A breaks the language out of the host. A Penelope program, written by a user, parsed by a Penelope-implemented parser, compiled by a Penelope-implemented WASM backend, runs in any environment that supports WebAssembly — Node, browsers, Cloudflare Workers, Wasmtime, the wasm runtime baked into your phone. The TS implementation is the bootstrap stage; once you have a WASM-targetable self-hosted frontend, the TS code becomes optional, not load-bearing.

The int-only subset is intentionally where the WASM story is cleanest: no GC, no host imports, no continuation transforms. Each excluded feature in the table above is a known path to extend, and each one is its own design conversation that deserves its own pass. The current ~360-line backend is the seed crystal that makes those next steps concrete instead of speculative.

What's next

  • Phase 6.B (potential): WASM target for strings + lists via a small allocator in linear memory. Would let the self-hosted parser/compiler itself emit to WASM and self-host on the WASM runtime.
  • Phase 6.C (potential): CPS transform → snapshot/pause in WASM, so the durable-execution model works in WASM environments (Cloudflare Workers being the obvious target).
